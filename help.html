PSX.Dev Visual Studio Code Extension


The PSX.Dev community have extracted the original Sony PSX SDK libraries and made them usable with a modern GCC compiler and debugger (called PSYQ + nugget ).
They are also maintaining the PS1 emulator PCSX-REDUX which also allows GDB to step through code. And they have combined all these projects into a simple Visual Studio code extension.

The Net Yaroze SDK being a reduced subset version of the professional PSYQ SDK, it is therefore relatively simple to create a Net Yaroze compatible project.

First install Visual Studio and the PSX.DEV extension by following the most excellent video instructions to get started.
   

PSX.Dev Visual Studio Code Extension for PS1 Homebrew development, by Nicolas Noble.

Follow the video and run the Psy-Q Cube template demo as per the video. Make sure it builds, runs and you can step debug correctly. Installing clangd is also recommended.

Installing Net Yaroze template.

This Net Yaroze template requires the Psy-Q and nugget tools, if you ran the Psy-Q Cube demo, there's nothing else to install.
The instructions are the same as per the Psy-Q Cube demo:
 
     
     
  1. In Visual Code, hit CTRL + Shift + P, select PSX.DEV Show panel, click the TEMPLATES tab.
  2. Enter the name and path of your project and click the Psy-Q SDK tab, then click create, in the Net Yaroze section, to create the new template project.
  3. The Net Yaroze template copies the Psy-Q SDK (libraries and headers) in the project folder, with the modified Net Yaroze headers and creates a demo program originally from here. It should automatically load the new project into Visual Code.
  4. Now we are ready to run and debug, Hit CTRL + Shift + P and select PSX.DEV: Build Debug, it should build correctly with no warnings or errors, and a XXX.ps-exe file, (where XXX is your project name) should have been created.
  5. Hit CTRL + Shift + P again and select PSX.Dev: Launch PCSX-Redux, this will launch the emulator but it will not start.
  6. Return to Visual Code and hit F5 to start the debugger, step over a few lines and loop through the while( !DONE ) loop, put a break point after the if( PAD_PRESS(buffer1,PAD_UP) ) line at sprite.y -=5;  about line 86.
  7. Next click run, to return to the step debugger, simply hit the up button on the controller.
  8. Close the emulator to stop debugging.

 

Found issues with Psy-Q Nugget library.

 

Hierarchical node parenting doesn't work.

Using the GsCOORDINATE2 field super which is used to set the parent node doesn't work. It will not display parented objects or display with screwed projection.
To fix it requires recreating the psy-q nugget libraries, a newly created libgs.a file is located in your third_party/net-yaroze folder, to use it, in your makefile just add this location as a library source, with the -L option like this:
 
 
LDFLAGS=-Xlinker --defsym=TLOAD_ADDR=0x80090000    -Lthird_party/net-yaroze 

Replicating the Net Yaroze Siocons script.

Siocons is a DOS terminal program which can send and receive data from the Net Yaroze after it has load the Net Yaroze boot disc.
It can mostly be automated by using a batch script file (typically .sio, .auto or .bat) which loads data files to the PlayStation at specific RAM addresses. Maintaining these addresses sequentially and saving space is a pain so it's recommended that the siocons script file be automatically generated using a tool like YOC which also creates a header file.

This siocons script is also used by yarexe to package all the files into a single PS-X executable file, which makes it easier to run in emulation, or send to a real PlayStation or burn etc.

In a real Net Yaroze project, you would load your files into RAM via the siocons script, in PSX.DEV the debugger has to pre-loaded the assets into the correct RAM address when pressing F5.

This is done via .vscode\launch.json in your project folder.
You would add your assets a similar way to the siocons script, ie:

 
local dload data\OPTIONS.TIM 0x801292E8
local dload data\SWORD.TIM 0x8012A508 
local load y3d.ps-exe
go

In the .vscode\launch.json file, the autorun section, the above equivalent would be: 
 "autorun": [
 "monitor reset shellhalt",
 "restore data/OPTIONS.TIM binary 0x801292E8",
"restore data/SWORD.TIM binary 0x8012A508",
 "load ${workspaceRootFolderName}.elf",
 "tbreak main",
 "continue"
 ]

YOC  can manage this list, all you have to do is insert your yoc_vscode.sio file into your .vscode/launch.json file like this:
 
 
   "autorun": [
   "monitor reset shellhalt",
     "source yoc_vscode.sio",
     "load ${workspaceRootFolderName}.elf",
     "tbreak main",
     "continue"
  ]

Reading Net Yaroze boot disc data.

The Net Yaroze boot disc has a few generic assets that can be load and used in your games.


And contrary to common belief, the Net Yaroze SDK can read data files from the CDROM.
On a real Net Yaroze PlayStation, it can only be official PlayStation CDROM's, which basically means the Net Yaroze boot disc, which does come with a few useful game ready assets. But with a chipped PlayStation or emulator it can read any PlayStation formatted CDROM.

Quick description of the file types:

.TIM  - Image format, after loading the TIM into VRAM, the RAM space can be reused.
.SEQ - Music sequence, similar to the midi format, and must stay in RAM to play.
.VB  - Voice Bank, sampled audio, after loading the VB file into Audio RAM, the RAM space can be reused.
.VH  - VB index information, must remain in RAM to work.
 
Location  About size: 
/DATA/FONTS/7SEGFONT.TIM Black and White, large font but limited characters. 256x128 8bit - 17KB
/DATA/FONTS/8DOTFONT.TIM Black and White, small font but with more character (kanji and font images). 256x128 8bit - 17KB
/DATA/MODEL/ The models are in RSD & DXF formats and not usable without being converted first to TMD on a PC!

AP01 - fighter jet - 480 faces
CAR - 2 door, red car (with individual parts)- 428 faces
PGIRL - purple lady with blonde hair - 1037 faces
PMAN - gray man with no hair - 948 faces

TMD's must stay in RAM.
/DATA/SOUND/GOGO.SEQ Loopable, sounds like: Terra Incognita - intro & game 1.20mins - 16.9KB
/DATA/SOUND/MUSI.SEQ Loopable, sounds like: Terra Incognita - last level 2.22mins - 38.7KB
/DATA/SOUND/SAMPLE1.SEQ The same file as MUSI.SEQ above
/DATA/SOUND/STD0.VB Sound Bank - Sound fonts instruments, sounds like Terra Incognita above. 391KB
/DATA/SOUND/STD0.VH Sound Header - Used to reference sound/instruments data in sound RAM. 59KB
/DATA/TEXTURE/FANCY 18 32x32, 16x16 and 96x96 8bit patterns
/DATA/TEXTURE/FIRE 4 frames of animation fire 1.53KB each frame
/DATA/TEXTURE/FRAME/FRAME01.TIM Small frame segments, with think and thin out lines, grey scale, 8bit 128x32 2.06KB
/DATA/TEXTURE/FRAME/FRAME02.TIM Large frame segments, think lines only, grey scale, 8bit, 128x64 4.06KB
/DATA/TEXTURE/FRAME/GREEN 2x Green grass and a Moss 8bit textures in both 64x64 and 32x32
/DATA/TEXTURE/FRAME/METAL 2x metal textures 8bit 32x32
/DATA/TEXTURE/FRAME/STONE 2x sand, 2x stone, 2x wall 8bit 32x32 and 64x64 textures
/DATA/TEXTURE/FRAME/TILE 2x net, 6x tile (TILE01,02 are tile sets) 8bit varying sizes
/DATA/TEXTURE/FRAME/WAVE 6x FALL (water fall) 3x WAVE water ripple)
/DATA/TEXTURE/FRAME/WOODY 7x wood grain textures in horizontal and vertical

Playing CD Digital Audio.

The Net Yaroze can also play any audio CD tracks( CD-DA - CD Digital Audio), and are played in the background without any CPU resources.
This can be done by using the CUE/BIN files instead of just the .iso image.


The debugger must first load the cdrom image, similar to the loading of the assets from the siocons script.
In the .vscode\launch.json file, the autorun section:
  
 "autorun": [
 "monitor reset shellhalt",
 "monitor mountcd Net-Yaroze-Boot-Disc.cue",
 "load ${workspaceRootFolderName}.elf",
 "tbreak main",
 "continue"
 ]
 

Currently, the PCSX-R emulator has an issue getting the CD-DA track listing, in Net Yaroze SDK this is done via:



// 3 = The index of the tracks array for the track currently being played is returned. (No other action is taken.)
CdPlay(3, 0, 0); 
 
Do not call CdPlay with the parameter 3, instead hardcode your total track number and manage the currently playing track manually.
Playing CD-DA tracks does work as expected.


For tutorials on reading CDROM data or playing CD digital audio see chapter 3.


Changing the linker settings.

Technically speaking, this step is not required as it doesn't require the Net Yaroze LibPS.exe in RAM for dynamic linking and running. The ps-exe files are statically link.
However if you want to be 100% completely Net Yaroze compatible (and not cheat!) this RAM space must not be used at all.

The Net Yaroze linker setting are defined typically like:
-Xlinker -Ttext -Xlinker 80090000

Where 0x80090000 is the RAM location that the Net Yaroze executable would be typically loaded to.

In PSX.DEV the default is 0x80010000 (PSX RAM address range is from 0x80010000 to 0x801FFFFF).
Loading at 0x80010000 is a problem for Net Yaroze programs because that's the location of the Net Yaroze libps.exe file (like a DLL which makes executables smaller to transfer over serial - See Memory Map in the Net Yaroze User Guide PDF).
We need to change it to the first RAM available to us, which is after libps.exe, 0x80090000 (loaded low).
To do this with PSX.DEV's GCC, add this line as the first LDFLAGS line in the makefile:
LDFLAGS=-Xlinker --defsym=TLOAD_ADDR=0x80090000

But the executable can be loaded anywhere in the PlayStation's RAM. For example the executable can be loaded after the assets or data files, loaded higher in RAM.

Please note: The Net Yaroze executable will always be alot smaller than a PSY-Q link executable which will be a lot bigger because it will include the linked libraries in the executable. This change will effect your siocons script IF your data is loaded after the executable (loaded low in RAM).
 

Reusing asset RAM.

If you only load your assets once in your program (which is typical for Net Yaroze) and you organize your files correctly, then after they have been copied to their respective places ie TIM's to video RAM and VB's to audio RAM and initialized correctly, their original RAM space can then be reclaimed by your program for anything, ie dynamically created game data etc or by initializing the heap (via InitHeap) to start at the top of this RAM space (which then grows upwards).

Obviously you can not reclaim RAM space of files that are still required to be in RAM, ie TMD's.
 

Using with the original Psy-Q.

This template can also be used with the original Sony Psy-Q toolchain.

Environment Batch file:
REM REM
===== PSX DEVELOPMENT ENVIRONMENT VARIABLES ==== RELEASE 1.8 LIBRARY 3.6.1 Date: 1-31-1998 Time: 21:38:44 set PATH=C:\Psyq\bin;
set PSX_PATH=C:\Psyq\bin
set LIBRARY_PATH=C:\Psyq\lib
set C_PLUS_INCLUDE_PATH=C:\Psyq\include
set C_INCLUDE_PATH=C:\Psyq\include
set PSYQ_PATH=C:\Psyq\bin
REM
REM
REM
REM ==
Does your computer lack a floating-point co-processor? If so, uncomment the following line:
set G032=emu d:\ps\bin\emu 387
REM EXAMPLE DEVICE DRVIERS. Modify according
REM to information in the "2000.doc" or "2500.doc" document REM in the Technical Reference CD.
REM For the DTL-H2000 (ISA) board:
REM d:\ps\bin\dexbios /a1340 /i11
REM For the DTL-H2500 (PCI) board (decicons only):
REM d:\ps\bin\h25drv.exe /v/n
REM For the CD-ROM emulator (ISA) board:
REM d:\ps\bin\cdbios /a388 /17/d6
set COMPILER_PATH=C:\Psyq\bin
set G032=DPMISTACK 1000000
set G032TMP=C:\WINDOWS\TEMP
set TMPDIR=C:\WINDOWS\TEMP

set PATH=C:\psyq\bin\;%PATH%
 

 

Running:
 
CCPSX -Ithird_party/net-yaroze/include  main.c  -g -Xo$80080000 -c -o car.obj
dumpobj  car.obj > car.mapfile
CCPSX -Ithird_party/net-yaroze/include  main.c  -g -Xo$80080000 -o car.cpe,car.sym,car.map
cpe2x car.cpe
no$psx car.exe

Tips & Notes.

 

Debugging problems.


#1 Check your path, make sure you are running the right tool and version! It should always be using the PSX.Dev toolchain supplied applications.
#2 Alway use the latest versions of the toolchain, libraries, emulator, etc.
 
 

Call GsInitGraph first!

Psy-Q requires the libGS libraries to be initialize with a GsInitGraph call FIRST before using other libGS functions, in particular FntLoad or GetPadBuf .
Modern GCC can be very strict, make sure all your functions that return a value, does return a value, this is especially true when debugging using GDB.
 

KanjiFntPrint

 
Both FntPrint and KanjiFntPrint prototypes were empty in libps.h this was incorrect and are correctly defined as:


int FntPrint(int id, char *, ...);
int KanjiFntPrint(int id, char *, ...);
 
 

Makefile.

C Language.


The default lanaguage is C++, if you want to use C, edit your makefile in your project like the following.

Above the line :
CPPFLAGS +=

Add the C standard you want, IE:
CFLAGS += -std=c99

And also rename:
CPPFLAGS
To just:
CFLAGS
 

Executabe extension.

 
While looking at the makefile, if you change the executable extention, the default is ps-exe:
TYPE = ps-exe
To something like:
TYPE = exe

You will also have to change the linker script filename located in:
\third_party\nugget\ps-exe.ld
You will need to change it to:
\third_party\nugget\exe.ld

Support.

The Psy-Q to Net Yaroze glue (psyq_to_netyaroze) is still Work In Progress.


There are a few things missing or not working, these are typically:

1) Compiler built-in functions (exit, abort, etc) - If you come across a C language function that doesn't work, test it in the Psy-Q Cube demo to verify it works first, if it doesn't work or is wrongly defined, then it could be a Psy-Q conversion issue;

2) or Net Yaroze only helper functions like TestCard, GetPadBuf which are not found in Psy-Q.
 

If you have problems, errors or fixes, please raise an issue here:

https://github.com/gwald/psyq_to_netyaroze/issues
